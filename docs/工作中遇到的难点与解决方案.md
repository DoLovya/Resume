# 工作中遇到的难点与解决方案

## 1. 现场环境附加进程调试断点失效问题

### 问题背景
在生产现场排查软件故障时，需要直接在现场电脑上对运行中的程序进行调试。通常采用 IDE（如 Visual Studio）的“附加到进程”（Attach to Process）功能接入目标程序。

### 遇到的困难
在调试过程中发现，设置在关键逻辑处的**部分断点无法触发**（无法命中），导致无法暂停程序执行以查看堆栈和变量信息，严重阻碍了问题的定位。

### 分析与排查
1.  **初步检查**：首先确认了 PDB 符号文件已正确加载，且与现场运行的二进制文件版本一致，排除了符号不匹配的问题。
2.  **深入分析**：考虑到现场运行的通常是 Release 版本，编译器会对代码进行优化（如内联、指令重排等）。这种优化可能导致生成的机器码与源代码的行号映射关系不再精确，或者部分代码逻辑被优化调整，从而导致调试器无法在预期的源码行上正确捕获断点。

### 解决方案
为了在不影响整体系统环境的前提下进行调试，我采取了**局部去优化**的策略：
1.  定位到疑似包含 Bug 的特定动态库模块。
2.  在构建系统中修改该模块的编译配置，**关闭编译优化**（例如在 MSVC 中设置 `/Od`，或在 CMake 中针对该 target 设置编译选项）。
3.  重新编译生成该特定模块的 Release 版本（仅去除优化），并替换现场的对应文件。

### 结果
替换文件后重新附加进程调试，断点能够正常触发。通过单步调试成功复现并定位了逻辑错误，最终解决了现场问题。

## 2. WPF 平台自定义字体引发的内存泄漏

### 问题背景
测试团队反馈“视觉应用开发平台”存在内存泄漏问题。泄漏速度缓慢（约几 KB/秒），但长期运行会导致内存显著增长。该问题此前已由同事尝试修复未果，转交由我继续攻关。

### 分析与排查
1.  **工具分析**：使用 Visual Studio 的内存性能分析器（Memory Profiler）对程序进行长时间挂机监控（运行数天），以积累足够的数据来捕捉微小的内存增长趋势。
2.  **定位热点**：分析内存快照，发现内存占用增长最多的是某些特定的 WPF 内部数据结构。
3.  **溯源研究**：针对该数据结构进行 Google 检索，最终在 GitHub 的 WPF 仓库 Issue 列表中发现类似报告。
4.  **根因确认**：确认为 WPF 框架层面的已知 Bug——当应用程序使用**自定义字体资源**（Pack URI 方式加载）时，会引发非托管内存泄漏。进一步测试证实，若使用系统已安装字体或绝对路径加载字体，则无此问题。

### 解决方案
鉴于这是框架底层的缺陷，且自定义字体非核心功能强需求，最终决定**移除自定义字体资源**，回退使用系统默认字体。

### 结果
去掉自定义字体后，经长时间压力测试验证，内存曲线平稳，泄漏问题彻底解决。
